# 画像仕様書

Swish Gallery ランディングページで使用する画像の仕様と要件をまとめたドキュメントです。

## 📁 ディレクトリ構造

```
src/
├── images/          # メインコンテンツ画像
│   ├── hero-main.png
│   ├── feature-grayscale.png
│   ├── feature-minimal.png
│   └── feature-board.png
└── assets/
    └── icons/       # アイコン・ロゴ
        ├── logo.svg
        ├── arrow-right.svg
        └── microsoft-store.svg
```

## 🖼️ 必要な画像一覧

### 1. **hero-main.png** (ヒーローセクション)

**用途**: トップページのメインビジュアル

**推奨仕様**:

- **アスペクト比**: 21:9 (超ワイド)
- **推奨解像度**: 2100×900px 以上 (Retina 対応: 4200×1800px)
- **ファイルサイズ**: 500KB 以下 (WebP 推奨、PNG でも可)
- **ローディング**: eager (即座に読み込み)

**内容**:

- Swish Gallery のメインインターフェースを表示
- 余計な枠線が一切ない、クリーンなウィンドウデザイン
- 高解像度の作品画像を表示している状態
- 背景はデスクトップ環境を想起させるクリーンな色
- ボーダーレスデザインの特徴が一目でわかる構図

**デザイン要件**:

- UI 要素を最小限に
- 作品に集中できる雰囲気
- プロフェッショナルかつモダンな印象

---

### 2. **feature-grayscale.png** (色価プレビュー機能)

**用途**: グレースケール変換機能の説明

**推奨仕様**:

- **アスペクト比**: 4:3
- **推奨解像度**: 1600×1200px 以上 (Retina 対応: 3200×2400px)
- **ファイルサイズ**: 400KB 以下
- **ローディング**: lazy

**内容**:

- カラー画像と色価を保ったグレースケール画像の比較
- 左右分割、またはビフォー・アフター形式
- 明度バランスの違いが明確にわかる作例
- (オプション) 単純な彩度 0 との比較も含めると効果的

**推奨作例**:

- 人物ポートレート、風景、イラストなど色情報が豊富な画像
- グレースケール変換の精度が際立つもの

---

### 3. **feature-minimal.png** (ボーダーレスデザイン)

**用途**: UI の透明性とボーダーレスデザインの訴求

**推奨仕様**:

- **アスペクト比**: 1:1 (正方形)
- **推奨解像度**: 1200×1200px 以上 (Retina 対応: 2400×2400px)
- **ファイルサイズ**: 300KB 以下
- **ローディング**: lazy

**内容**:

- デスクトップ背景の上に浮かぶような、枠のないウィンドウ
- タイトルバー、メニューバー、ボーダーが一切ない状態
- 画像の比率に吸着した美しいレイアウト
- 背景はクリーンで落ち着いたトーン (グレーまたはベージュ系)

**デザイン要件**:

- ウィンドウ枠が「ない」ことを強調
- デスクトップとの調和を表現
- シームレスな体験の視覚化

---

### 4. **feature-board.png** (リファレンスボード機能)

**用途**: 複数画像を自由配置できるボード機能の紹介

**推奨仕様**:

- **アスペクト比**: 16:8 (2:1)
- **推奨解像度**: 2400×1200px 以上 (Retina 対応: 4800×2400px)
- **ファイルサイズ**: 600KB 以下
- **ローディング**: lazy

**内容**:

- 複数の画像がコラージュのように自由に配置されている様子
- 画像のサイズや角度がバラバラでも整然とした印象
- クリエイターが資料を並べて作業している雰囲気
- 背景はキャンバスを思わせるテクスチャまたは無地

**推奨構成**:

- 3〜6枚程度の画像を配置
- スケッチ、参考写真、カラーパレットなどが混在
- ドラッグ&ドロップで配置した直後のような自然な配置

## 🔧 画像最適化

### 推奨フォーマット

1. **WebP** (最優先): 高圧縮率、透過対応
2. **PNG**: 品質重視、透過が必要な場合
3. **JPEG**: 写真など、透過不要な場合

### 最適化ツール

- [Squoosh](https://squoosh.app/) - Google 製の画像圧縮ツール
- [TinyPNG](https://tinypng.com/) - PNG/JPEG の圧縮
- [ImageOptim](https://imageoptim.com/) (Mac) - バッチ最適化

### パフォーマンス要件

- **LCP (Largest Contentful Paint)**: 2.5秒以内
- **hero-main.png** は特に重要 → 500KB 以下を厳守
- レスポンシブ対応のため、必要に応じて複数サイズを用意

---

## 📱 レスポンシブ対応 (オプション)

モバイル表示の最適化のため、以下のサイズも用意可能:

```
images/
├── hero-main.png          # デスクトップ (2100×900)
├── hero-main@mobile.png   # モバイル (750×320)
├── feature-grayscale.png
├── feature-grayscale@mobile.png
...
```

HTML 側で `<picture>` タグを使用し、デバイスに応じた画像を配信できます。

## 📝 補足: OGP 画像 (SNS シェア用)

現在 HTML に記載されている OGP 画像も作成が必要です:

**ファイル名**: `ogp.png`  
**配置先**: `src/images/ogp.png`  
**推奨仕様**:

- **サイズ**: 1200×630px (Facebook/Twitter 推奨)
- **ファイルサイズ**: 300KB 以下
- **内容**: ロゴ + キャッチコピー + アプリのスクリーンショット

画像配置後、以下を確認してください:

1. ブラウザで表示確認 (Chrome DevTools のデバイスモード)
2. Lighthouse でパフォーマンススコアを測定
3. 画像の遅延読み込みが正常に動作しているか
4. プレースホルダーが適切に表示されるか
